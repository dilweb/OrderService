# План тестирования Order Service

## Цель
Понять, как тестируют код на проде: от теории к практике. Начинаем с **юнит-тестов** на **pytest** и постепенно покроем проект.

---

## 1. Теория: зачем и как тестируют

### Зачем тестировать
- **Регрессии**: после изменений старый функционал не ломается.
- **Документация**: тесты показывают, как код должен себя вести.
- **Уверенность**: можно рефакторить и деплоить без страха.

### Уровни тестов (пирамида)

```
        /\
       /  \   E2E (редко, медленно)
      /----\
     /      \  Интеграционные (API + БД/Kafka)
    /--------\
   /          \  Юнит-тесты (много, быстро)
  /------------\
```

- **Юнит-тесты** — проверяют одну функцию/класс в изоляции, без БД, Kafka, HTTP. Быстрые, много.
- **Интеграционные** — проверяют связку (эндпоинт + БД, или producer + Kafka). Медленнее, нужны тестовые сервисы.
- **E2E** — проверяют весь сценарий от запроса до ответа в «почти продовой» среде.

На проде **не запускают** тесты — их запускают в CI (GitHub Actions, GitLab CI и т.д.) перед деплоем. Прод только получает уже проверенный артефакт.

## 2. Юнит-тесты с pytest: основы

### Установка
```bash
cd OrderService
poetry add --group dev pytest pytest-asyncio
```

### Структура теста: AAA
- **Arrange** — подготовить данные (входы, моки).
- **Act** — вызвать тестируемую функцию.
- **Assert** — проверить результат.

### Пример
```python
def test_calculate_total_one_item():
    # Arrange
    order = OrderCreate(
        user_id=1,
        items=[OrderItem(product_id=1, name="Tea", price=Decimal("10.50"), quantity=2)],
        currency="USD",
        amount=Decimal("21.00"),
    )
    # Act
    total = order.calculate_total()
    # Assert
    assert total == Decimal("21.00")
```

### Полезные возможности pytest
- `assert x == y` — не нужен `self.assertEqual`, достаточно `assert`.
- **Параметризация**: один тест — много входов:
  ```python
  @pytest.mark.parametrize("price,quantity,expected", [(10, 2, 20), (5, 3, 15)])
  def test_calc(price, quantity, expected):
      ...
  ```
- **Фикстуры** (`@pytest.fixture`) — переиспользуемая подготовка (объекты, моки).
- **Моки** (`unittest.mock` или `pytest-mock`) — подмена Kafka, БД и т.д., чтобы тест не ходил в реальные сервисы.

## 3. Что тестируем в Order Service (план по шагам)

### Шаг 1 — Модели заказа (самое простое)
- **Файл**: `src/orderservice/models/order.py`
- **Функция**: `OrderCreate.calculate_total()` — чистая логика, без I/O.
- **Ещё**: валидация Pydantic (например, пустой `items`, отрицательная цена).

**Как тестируют на проде**: эти тесты просто проверяют формулу и контракт моделей; они не зависят от окружения и всегда выполняются в CI.

### Шаг 2 — Kafka consumer (десериализация)
- **Файл**: `src/orderservice/kafka/consumer.py`
- **Функция**: `OrderKafkaConsumer._deserialize(value: bytes)` — статический метод, на входе bytes, на выходе dict.
- **Кейсы**: валидный JSON, пустой `value`, невалидный JSON.

**Как тестируют на проде**: юнит-тест не подключается к Kafka, только проверяет преобразование bytes → dict.

### Шаг 3 — Эндпоинт POST /orders (с моком Kafka)
- **Файл**: `src/orderservice/api/routers.py`
- **Функция**: `create_order` — проверка `amount` vs `calculate_total()`, возврат 202/422/503.
- **Как**: через `TestClient` FastAPI + мок `get_producer()`, чтобы не слать события в реальный Kafka.

### Шаг 4 — Интеграционные тесты (позже)
- Реальный Kafka в docker-compose для тестов или Testcontainers.
- Проверка «запрос пришёл → событие появилось в топике».

## 5. Запуск тестов

```bash
cd OrderService
poetry run pytest tests/ -v
```

- `-v` — подробный вывод.
- `-k "test_calculate"` — только тесты, в имени которых есть `test_calculate`.
- `--tb=short` — короткий traceback при падении.

После каждого шага запускай тесты и смотри, что они зелёные — так ты закрепишь, как это делается на проде (в CI тесты запускаются так же, только без интерактива).
